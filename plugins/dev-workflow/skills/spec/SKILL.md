## 目的

会話中の仕様サマリー（grill-meの出力）をもとに、2つのドキュメントを `docs/specs/` に作成する。

## 手順

### 1. 仕様サマリーの確認

会話履歴から仕様サマリーを探す。見つからない場合はユーザーに確認する。

### 2. コードベースの調査

ドキュメント作成前に、既存のコードベースを調査する:
- プロジェクトの技術スタック・構成を把握する
- 関連する既存コードを確認する
- 命名規則やパターンを理解する

### 3. 仕様書の作成

`docs/specs/$ARGUMENTS/spec.md` に以下の構成で作成:

```markdown
# 仕様書: [機能名]

## 1. 概要
[目的と背景]

## 2. ユーザーストーリー
[ユーザー視点の要件]

## 3. 機能要件
### 3.1 [要件グループ]
- [要件詳細]

## 4. 非機能要件
- パフォーマンス
- セキュリティ
- アクセシビリティ

## 5. データモデル
[スキーマ設計]

## 6. 画面設計
[UI構成・コンポーネント]

## 7. API / インターフェース
[モジュール間通信]

## 8. スコープ外
[除外事項]
```

### 4. 実装計画書の作成

`docs/specs/$ARGUMENTS/plan.md` に以下の構成で作成:

```markdown
# 実装計画書: [機能名]

## 概要
[実装のゴール]

## 前提条件
[既存コードへの依存]

## タスク一覧

### Phase 1: データ層
- [ ] Task 1: [タスク名]
  - 対象ファイル: [パス]
  - 内容: [詳細]
  - 完了条件: [テストが通る等]

### Phase 2: ビジネスロジック
- [ ] Task 2: ...

### Phase 3: UI / プレゼンテーション層
- [ ] Task 3: ...

### Phase 4: 統合・結合
- [ ] Task 4: ...

### Phase 5: テスト・仕上げ
- [ ] Task 5: ...

## 依存関係
[タスク間の依存]

## リスク・注意点
[実装時の注意事項]
```

### 5. 完了案内

ドキュメント作成後、以下を案内する:
- 「`/dev-workflow:epic $ARGUMENTS` でGitHub issueを作成できます」
